# Butler agent learnings and architecture guide
## Learnings from implementation iterations
- Prefer plain message content for RSVP updates when users want less visual noise; this keeps edits deterministic and avoids embed layout churn.
- Discord controls URL unfurl behavior and image preview sizes, so UI requests around image sizing/positioning often need product-level compromises (emoji + text is the most stable compact option).
- If an event embed should appear above editable RSVP content, post the event URL as a separate message first, then post the editable RSVP message with buttons/reactions.
- Reaction precedence must be explicit when multiple emojis are present to avoid nondeterministic status assignment.
- Keep fallback behavior for optional cosmetic assets (edition emoji/logo): missing visual assets should never block posting or editing.

## Architecture overview
### Core modules
- `butler/app.py`
  - Discord bot entrypoints, slash commands, permission checks, and event/reaction handlers.
  - Orchestrates creation and posting lifecycle for scheduled events and RSVP messages.
- `butler/rsvp_view.py`
  - Interactive RSVP view (`discord.ui.View`) with buttons/modals and content renderer.
  - Owns in-memory RSVP state for a specific posted message.
- `butler/rsvp_domain.py`
  - Typed domain model for RSVP statuses and pure helpers for counts/mentions/state transitions.
- `butler/design.py`
  - Presentation constants and message template strings.
- `butler/event_logic.py`
  - Event input normalization/validation (times, room URL, defaults).
- `butler/permissions.py`, `butler/discord_helpers.py`, `butler/settings_store.py`
  - Permission abstraction, Discord fetch helpers, and persisted guild config.

### Runtime state model
- `ACTIVE_RSVP_VIEWS` in `butler/app.py` maps `message_id -> AvailabilityView`.
- The RSVP message is the editable source of truth for visible state.
- Status data is in-memory (session scoped); reaction add/remove events recompute and rewrite message content.

## Message posting flow
### `/eventtoday`
1. Validate input and permissions.
2. Create scheduled event.
3. Post event URL message first (for Discord event embed unfurl).
4. Post editable RSVP message with buttons/reactions.
5. Track RSVP message in `ACTIVE_RSVP_VIEWS`.

### `/previeweventdesign`
- Posts only the editable RSVP preview message (no scheduled-event creation).

## RSVP rendering model
- RSVP content is plain text generated by `AvailabilityView.build_content`.
- Edition line is rendered as emoji + edition name (no link) when an emoji is available.
- Room line states:
  - Closed: `🚪 **RUMMET ÖPPNAS NÄR LÄNKEN ÄR KLAR**`
  - Open: `**Rummet är öppet:** <url>`
- Status sections are generated from typed domain data and include counts + mentions.

## Reaction and button behavior
- Supported statuses: `Available`, `Maybe`, `Later`, `Storyteller`.
- Buttons set statuses directly.
- Reaction mapping priority is explicit:
  1. `🤔` -> `Maybe`
  2. `🕒` -> `Later`
  3. `😈` -> `Storyteller`
  4. any other emoji -> `Available`
- On reaction removal, remaining reactions are recomputed using the same precedence.

## Extending safely
- Add new status by updating all of:
  - `RsvpStatus` in `butler/rsvp_domain.py`
  - `RSVP_STATUS_EMOJIS` + labels in `butler/design.py`
  - button handler(s) in `butler/rsvp_view.py`
  - reaction mapping/recompute logic in `butler/app.py`
- Keep message template placeholders aligned with `build_content` format arguments.
- Keep user-facing copy centralized in `butler/design.py` to reduce accidental string drift.
